# User Skill Service - API Contract

## Design Overview

**Service Name**: user_skill_service
**Protocol**: HTTP REST API
**Parent Epic**: xenoISA/isA_#183
**Source Issue**: xenoISA/isA_user#341
**Status**: Contract ready for implementation

The user skill service stores user-created automations that can be managed from
the isA_ SkillBuilder UI and later exposed to Mate/runtime execution. The
service is the durable source of truth for authenticated users; clients may keep
an offline mirror, but local cache must not be treated as synced state until the
backend accepts the change.

## Ownership And Auth

- Every skill belongs to exactly one `owner_user_id`.
- All endpoints require an authenticated user context.
- Clients may pass `user_id` only where existing isA_user gateway conventions
  require it; the service must validate that it matches the authenticated user.
- A user can list, create, update, enable, disable, and delete only their own
  skills.
- Ownership mismatch returns `403 forbidden`; missing auth returns `401
  unauthenticated`.
- Skill identifiers are opaque stable ids generated by the backend.

## Data Model

### UserSkill

| Field | Type | Required | Notes |
| --- | --- | --- | --- |
| `id` | string | yes | Opaque backend id, recommended prefix `usk_` |
| `owner_user_id` | string | yes | Authenticated owner |
| `name` | string | yes | 1-80 characters |
| `description` | string | no | 0-500 characters |
| `trigger_phrase` | string | yes | Normalized for uniqueness per owner |
| `steps` | UserSkillStep[] | yes | Ordered, non-empty |
| `enabled` | boolean | yes | Defaults to `true` |
| `version` | integer | yes | Starts at 1, increments on update |
| `created_at` | string | yes | ISO 8601 UTC |
| `updated_at` | string | yes | ISO 8601 UTC |
| `last_used_at` | string | no | ISO 8601 UTC |

### UserSkillStep

| Field | Type | Required | Notes |
| --- | --- | --- | --- |
| `id` | string | response | Stable backend id within the skill version |
| `type` | string | yes | `prompt`, `tool_call`, `handoff`, `condition`, or `delay` |
| `title` | string | yes | User-visible step label |
| `config` | object | yes | Type-specific JSON payload |
| `position` | integer | yes | Zero-based ordering |

### CreateUserSkillRequest

```json
{
  "name": "Summarize this thread",
  "description": "Collect context and produce a concise summary.",
  "trigger_phrase": "summarize this",
  "steps": [
    {
      "type": "prompt",
      "title": "Summarize",
      "config": {
        "prompt": "Summarize the current conversation in 5 bullets."
      },
      "position": 0
    }
  ],
  "enabled": true
}
```

### UpdateUserSkillRequest

All fields are optional, but at least one mutable field must be present.

```json
{
  "name": "Summarize current thread",
  "description": "Create a concise handoff summary.",
  "trigger_phrase": "summarize thread",
  "steps": [
    {
      "id": "step_1",
      "type": "prompt",
      "title": "Summarize",
      "config": {
        "prompt": "Create a concise handoff summary."
      },
      "position": 0
    }
  ],
  "enabled": false,
  "expected_version": 2
}
```

`expected_version` is optional. When provided, the service rejects stale updates
with `409 version_conflict`.

## REST API

Base path: `/api/v1/user-skills`

| Method | Path | Purpose |
| --- | --- | --- |
| `GET` | `/api/v1/user-skills` | List the authenticated user's skills |
| `POST` | `/api/v1/user-skills` | Create a skill |
| `GET` | `/api/v1/user-skills/{skill_id}` | Fetch one owned skill |
| `PUT` | `/api/v1/user-skills/{skill_id}` | Update a skill |
| `DELETE` | `/api/v1/user-skills/{skill_id}` | Delete a skill |
| `POST` | `/api/v1/user-skills/{skill_id}/enable` | Enable a skill |
| `POST` | `/api/v1/user-skills/{skill_id}/disable` | Disable a skill |
| `GET` | `/api/v1/user-skills/runtime/enabled` | Runtime-facing enabled skills |

### List Skills

`GET /api/v1/user-skills?limit=50&offset=0&enabled=true&query=summary`

Response:

```json
{
  "skills": [],
  "count": 0,
  "limit": 50,
  "offset": 0
}
```

### Create Skill

`POST /api/v1/user-skills`

Response: `201 Created`

```json
{
  "skill": {
    "id": "usk_123",
    "owner_user_id": "usr_123",
    "name": "Summarize this thread",
    "description": "Collect context and produce a concise summary.",
    "trigger_phrase": "summarize this",
    "steps": [],
    "enabled": true,
    "version": 1,
    "created_at": "2026-05-05T00:00:00Z",
    "updated_at": "2026-05-05T00:00:00Z"
  }
}
```

### Update, Enable, Disable, Delete

Update returns `200 OK` with the updated `skill`. Enable and disable are
specialized updates that return the same response shape. Delete returns:

```json
{
  "success": true,
  "skill_id": "usk_123"
}
```

## Error Contract

All non-2xx errors return a stable machine-readable code:

```json
{
  "error": {
    "code": "duplicate_trigger_phrase",
    "message": "A skill with this trigger phrase already exists.",
    "field": "trigger_phrase"
  }
}
```

Required codes:

| HTTP | Code | Meaning |
| --- | --- | --- |
| 400 | `validation_error` | Invalid name, trigger phrase, or steps |
| 401 | `unauthenticated` | Missing or invalid authenticated user |
| 403 | `forbidden` | Authenticated user does not own the skill |
| 404 | `not_found` | Skill does not exist for this owner |
| 409 | `duplicate_trigger_phrase` | Normalized trigger phrase already exists for owner |
| 409 | `version_conflict` | `expected_version` does not match current version |
| 500 | `internal_error` | Unexpected service failure |

## Storage And Migration Plan

Create schema/table:

```sql
CREATE SCHEMA IF NOT EXISTS user_skills;

CREATE TABLE user_skills.skills (
  id VARCHAR(64) PRIMARY KEY,
  owner_user_id VARCHAR(128) NOT NULL,
  name VARCHAR(80) NOT NULL,
  description TEXT,
  trigger_phrase VARCHAR(200) NOT NULL,
  trigger_phrase_normalized VARCHAR(200) NOT NULL,
  steps JSONB NOT NULL,
  enabled BOOLEAN NOT NULL DEFAULT TRUE,
  version INTEGER NOT NULL DEFAULT 1,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  last_used_at TIMESTAMPTZ
);

CREATE UNIQUE INDEX uq_user_skills_owner_trigger
  ON user_skills.skills(owner_user_id, trigger_phrase_normalized);

CREATE INDEX idx_user_skills_owner_enabled
  ON user_skills.skills(owner_user_id, enabled);
```

Migration sequence:

1. Add schema/table/indexes with no backfill.
2. Deploy API in read/write mode.
3. Keep isA_ local cache as an explicit offline mirror until the UI migrates.
4. Add runtime read path from `/runtime/enabled`.

## Runtime Exposure

Mate/runtime should use `GET /api/v1/user-skills/runtime/enabled` to load only
enabled skills for the authenticated user. Runtime responses may omit fields
that are not needed for execution, but must include `id`, `trigger_phrase`,
`steps`, and `version`.

Runtime invocation should record `last_used_at` asynchronously after successful
execution. Invocation metrics and execution history are out of scope for this
contract and should be tracked separately if needed.

## Implementation Notes

- Normalize trigger phrases by trimming, lowercasing, and collapsing internal
  whitespace.
- Preserve step order by sorting on `position` before persistence and response.
- Validate `steps` as non-empty for create and for any update that provides
  steps.
- Do not silently accept duplicate triggers; return `409
  duplicate_trigger_phrase`.
- Do not expose skills across organizations unless a later contract adds
  organization scope.
